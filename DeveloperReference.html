System Overview

How PlayArc, Project Manager, the REST API, and the Playwright runner connect

💡
What is PlayArc? A browser-based automation testing platform. It provides a visual UI to build, manage, and run Playwright test suites — without writing code. It imports existing Katalon Studio projects and executes them against any environment.

Run Lifecycle

Phase 1 — GENERATE  (server, synchronous)
  Write keyword library .ts files from DB
  Write objectRepo.ts helper from DB
  Write fixtures.ts listener file from DB
  Generate SuiteName-runId.spec.ts

Phase 2 — EXECUTE   (child process, async)
  spawn('npx', ['playwright', 'test', specPath, '--reporter=json'])
  Parse stdout events → broadcast() to all WebSocket clients
  On selector failure → self-heal via fallback chain

Phase 3 — REPORT    (server, on process exit)
  Parse results.json from Playwright output
  UPDATE runs SET status, passed, failed, duration
  Broadcast 'complete' event to UI

Key Design Decisions

DecisionRationale
Two separate HTML filesProject Manager handles project lifecycle; PlayArc handles execution. Each file is smaller and loads faster.
localStorage as project storeZero server dependency for project metadata. Works offline. Projects and environments persist across browser sessions without a database.
202 Accepted on POST /runRuns are async — respond immediately, stream progress via WebSocket. UI never polls.
SQLite for run historyZero config, file-based. Swap to Postgres by changing the DB layer only.
Spec files written to diskPlaywright requires real .ts files. Also gives developers transparent, readable generated output they can version-control.
Healscore at import timeBrittle selectors are flagged before the first run so developers fix them proactively, not reactively after CI failures.

Two-App Architecture

How PlayWriteDemo.html and ProjectManager.html work together via localStorage

PlayWriteDemo.html — The Execution UI

The primary test execution interface. Contains the Builder, Object Repository, Keyword Library, Listeners, Runner, and Reports. On startup it reads the active project from localStorage and hydrates all APP state. Users can also reload manually with the 🔄 Reload Project button in the sidebar without a page refresh.

Startup Sequence

window.addEventListener('load', () => {
  // 1. Hydrate from Project Manager's active project
  const p = JSON.parse(localStorage.getItem('playarc_current'));
  if (p) {
    if (p.suites?.length)      APP.suites      = p.suites;
    if (p.objRepo?.length)     APP.objRepo     = p.objRepo;
    if (p.dataTables?.length)  APP.dataTables  = p.dataTables;
    if (p.keywordLibs?.length) APP.keywordLibs = p.keywordLibs;
    if (p.listeners?.length)   APP.listeners   = p.listeners;
    // update topbar project name and logo
  }
  // 2. Sync environment names + LED colours from Project Manager
  syncEnvUI();
  // 3. Normal init
  loadSuite(0); refreshDashboard(); updateCounts();
});

Sidebar Integration Points

The sidebar bottom section exposes three integration controls:

  • Active Project pill — shows the current project name (cyan), updates whenever loadProjectFromStorage() is called
  • 📁 Project Manager ↗ — opens ProjectManager.html in a new tab
  • 🔄 Reload Project — re-reads playarc_current and live-updates all APP state and rendered pages without a page refresh

ProjectManager.html — The Manager

Handles everything before a test can run: creating projects, importing from Katalon, switching between projects, and configuring environments. Has no connection to the backend server — it is purely a localStorage management tool.

Generated objectRepo.ts

// Selector priority: data-testid → aria-label → css → xpath
const REPO = {
  'LoginDialog/btnLogin': {
    sel: '', aria: '[aria-label="login"]',
    css: 'button.btnPrimary',
    xpath: "(//button[@type='button'])[3]",
    healscore: 70
  },
  'MenuPage/SidesMenu/lnkMenu': {
    sel: '', aria: '[aria-label="menu"]',
    css: '',
    xpath: "//div[@id='app']/div/header/div/div[2]/a[2]",
    healscore: 68
  }
};

export function obj(page: Page, key: string) {
  const e = REPO[key];
  if (!e) throw new Error(`Object not found: ${key}`);
  if (e.sel)  return page.locator(e.sel);   // 1: data-testid
  if (e.aria) return page.locator(e.aria);  // 2: aria-label
  if (e.css)  return page.locator(e.css);   // 3: CSS
  return page.locator(e.xpath);              // 4: XPath
}

Builder & Steps

How test suites are constructed and edited in PlayArcDemo

Suite Operations

Suites are stored in APP.suites[]. The active index is APP.suiteIdx. loadSuite(idx) renders the step list and updates the detail panel and the Runner sidebar. New suites are created via + Suite in the topbar or imported from the Project Manager.

Step Editing

Steps are drag-and-drop reorderable via HTML5 drag events (APP.dragSrc tracks the dragged index). Each step card shows a coloured action tag, description, and status dot. Selecting a step populates the detail editor on the right — action type, description, value, and selector fields. The Playwright Preview pane below the editor live-renders the exact Playwright line that would be generated for that step.

Object Repo Quick Reference

The Builder's right sidebar shows a grouped object browser. Clicking an object inserts its best selector into the current step's selector field and sets step.objId. Objects flagged brittle show an orange ⚠ healscore badge so developers can choose a more stable selector before running.

Data Binding in Steps

Any step's value field may contain {{columnName}} tokens. At runtime the runner replaces them with values from the bound data table row. The Builder shows a live preview substitution when a data table is selected in the Runner.

Runner & Execution

How the UI executes suites and streams live results

execRun() flow

1.  POST /suites/:id/run  →  202 { runId }
2.  Open WebSocket: ws://localhost:4000/ws?runId=…
3.  For each step (simulated in demo mode):
    renderStepProg(suite, i)    // update step list dot colours
    updateBM(step, 'run')       // update browser mockup panel
    addLog('STEP', description) // append to streaming log panel
4.  On brittle selector: POST /heal → try fallback chain
5.  On complete: push run to APP.runHistory, refreshDashboard()

Self-Heal in the Runner

If a step's selector fails, the runner calls POST /heal with the primary selector and an array of fallbacks ordered aria → css → xpath. The server tries each fallback on the live page. On success it broadcasts a heal WebSocket event — the UI logs an orange HEAL line, increments the heal counter in the sidebar, and the run continues uninterrupted.

Environments

How the active environment is selected in PlayArc and how it is configured in the Project Manager

Topbar Environment Badge

The coloured LED + name badge in the top-right corner of PlayArc shows the active environment. Clicking it calls cycleEnv() which advances APP.envIdx through the array and calls syncEnvUI() to update the LED colour and label from playarc_envs.

syncEnvUI()

function syncEnvUI() {
  const envObjs = JSON.parse(localStorage.getItem('playarc_envs')) || [];
  if (envObjs.length) {
    APP.envs       = envObjs.map(e => e.name);
    APP._envColors = envObjs.map(e => e.color || 'var(--green)');
    if (APP.envIdx >= APP.envs.length) APP.envIdx = 0;
  }
  document.getElementById('envLbl').textContent       = APP.envs[APP.envIdx];
  document.getElementById('envLed').style.background = APP._envColors[APP.envIdx];
}

This function is called at startup, after cycleEnv(), and after loadProjectFromStorage(). Because it reads from localStorage, any changes made in the Project Manager's Environment panel are reflected in PlayArc immediately on the next cycle or page load.

Katalon Import Engine

How a Katalon Studio ZIP is parsed into a full PlayArc project — entirely in-browser

✅
No server or external libraries required. The import engine uses the File API and a hand-written ZIP central-directory parser. Uncompressed (stored) entries are decoded fully. Deflated entries are attempted via TextDecoder. Most Katalon ZIPs use stored compression for XML and Groovy files.

4-Step Wizard

StepWhat happens
1 — UploadDrag-and-drop or file picker. Accepts .zip files only. Displays import category overview cards (Objects, Suites, Keywords, Data Files, Profiles, Selector Healing).
2 — AnalyseZIP is parsed in memory. Files are categorised. All six parsers run in sequence. A live progress log and progress bar stream to screen.
3 — ConfigureSummary cards (object count, suite count, keyword count, data file count, brittle selector count). User sets project name (auto-filled from .prj), base URL, default browser, conflict mode. Full file table with healscore per object.
4 — DoneProject saved to playarc_projects. Profile environments merged into playarc_envs. Active project set automatically. Summary of imported counts shown.

File Parsers

parseZip() — ZIP Central Directory Reader

Reads the End-of-Central-Directory record to locate the central directory, then iterates all file entries. Uncompressed (method 0) files are decoded with TextDecoder. Directory entries are skipped. Returns an array of { name, content } objects covering all files in the archive.

parseRsFile() — Object Repository (.rs)

Reads Katalon .rs XML. Extracts: object name from <n>, page folder from file path, <selectorMethod> (which selector Katalon designated as primary), all selector entries from <selectorCollection> (XPATH / CSS / BASIC), aria-label from <webElementProperties> blocks, href and text content properties, and the xpath:idRelative / xpath:href webElementXpaths entries. Healscore is calculated from these properties — see algorithm below.

parseGroovy() — Test Scripts (Scripts/**/*.groovy)

  • Builds a varMap upfront by scanning for varName = WebUI.getText(findTestObject(...)) assignments, so verifyMatch(var, expected) can later resolve which object the variable came from.
  • Single-quoted bare strings ('Click Login') on their own line are Katalon step labels — the parser pairs each one with the WebUI.* call on the following line via a backward scan.
  • WebUI.openBrowser('') and WebUI.closeBrowser() are skipped and replaced by beforeSuite/afterSuite listener hooks.
  • Object names are resolved by matching the full Page/Name path from findTestObject() against the parsed object list using matchObjName().
  • A screenshot step is appended automatically to every suite.
  • Handles: navigate, click, fill, setText, setEncryptedText, getText, verifyMatch, verifyElementVisible, verifyElementNotVisible, delay, waitForElementVisible, callTestCase, selectOptionByLabel/Value/Index, executeJavaScript, switchToFrame, and any other WebUI.* as a custom step.

parseCustomKeywords() — Custom Keyword Library (Libs/*.groovy)

Scans for @Keyword-annotated methods. Extracts method name, typed parameters, and JSDoc description. The method body is scanned (brace-depth matching) and converted to a TypeScript comment stub with the original Groovy lines preserved as comments. All keywords from a single file are grouped into one KeywordLib entry.

parseGlbl() — Execution Profiles (Profiles/*.glbl)

Supports both the newer <GlobalVariableEntity>…<initValue> format and the older bare <n>/<value> pair format. Each profile becomes an Environment object. Colour is assigned based on the profile name (default → green, staging → orange, production/prod → red, dev/development → cyan, others → purple). The default profile is marked active: true.

parseCsvFile() — Data Files (.csv)

First row is treated as column headers. Subsequent rows become the data matrix. Handles quoted values and trims whitespace. Produces a DataTable with cols[] and rows[][].

Project Name (.prj)

The Katalon project file (*.prj) contains a <n> element with the canonical project name. The import engine reads this and pre-fills the project name field in Step 3 — more accurate than inferring the name from the ZIP filename.

Healscore Algorithm

Base score: 40

Bonuses:
  +30  aria-label property present            (most stable — semantically driven)
  +35  data-testid in any selector            (purpose-built test hook)
  +20  id-based XPath  //tag[@id='...']       (id attributes rarely change)
  +15  href attribute present                 (links are stable by design)
  +12  stable CSS — no child combinators, ≤5 class segments
  +10  role attribute in selector             (ARIA role semantics)
  + 8  short text content property (≤30 chars)

Penalties:
  -25  3+ positional indices [1][2][3]… in XPath  (highly brittle)
  - 6  per positional index up to 3
  -15  preceding:: or following:: neighbour XPaths
  -10  BASIC selectorMethod with normalize-space() pattern
  - 8  CSS with >6 chained class segments

Clamped: max(10, min(95, rawScore))
Objects with healscore < healThresh (default 50) → flagged as brittle
Brittle object names stored in project.brittle[] for UI highlighting

bestSelector() Priority

function bestSelector(obj) {
  if (obj.aria)  return obj.aria;                         // 1st: aria-label
  if (obj.href)  return `a[href="${obj.href}"]`;         // 2nd: href
  if (obj.css && obj.css.length < 80
      && !obj.css.includes(' > ')
      && obj.css.split('.').length <= 5)
    return obj.css;                                        // 3rd: compact CSS
  return obj.xpath || obj.basic || obj.primarySelector;   // 4th: XPath
}

Backend API

Self-Heal & Inspect

Automatic selector recovery and live element capture

POST/api/healTry fallback selectors against a live page
Body
{ selector: string, fallbacks: string[], url: string }
Response 200 — healed
{ healed: true, newSelector: "[aria-label='login']", strategy: "aria" }
Response 422 — failed
{ healed: false, tried: 3, error: "No fallback resolved" }
POST/api/inspectCapture element selectors from coordinates on a live page
Body
{ url: string, x: number, y: number }
Response 200
{ sel, aria, css, xpath, tag, text, screenshot }

Self-Heal Flow

1.  Playwright step throws — selector not found on page
2.  Runner catches error before marking step failed
3.  POST /heal { selector: primary, fallbacks: [aria, css, xpath], url }
4.  Server opens page in headless Chromium
5.  For each fallback: tries page.locator(fb).count() > 0
6.  Returns first resolving fallback with strategy label
7.  Run continues with healed selector — step marked pass
8.  WS broadcasts { type:'heal', original, healed, strategy }
9.  UI logs orange HEAL line, increments sidebar heal counter
10. PUT /objects/:id { [strategy]: healed } — persists for next run

WebSocket Protocol

Real-time run log and step status streaming

Connection

// Connect immediately after POST /suites/:id/run returns 202
const { runId } = await fetch(...).then(r => r.json());
const ws = new WebSocket(`ws://localhost:4000/ws?runId=${runId}`);
ws.onopen    = () => console.log('connected');
ws.onmessage = ({ data }) => handleEvent(JSON.parse(data));
ws.onerror   = (e) => console.error('WS error', e);
ws.onclose   = () => console.log('stream closed');

Server → Client Events

connected
On socket open
{ type: "connected", runId, ts }
status
Run state change
{ type: "status", status: "running"|"stopped", ts }
log
Every log line
{ type: "log", level: "INFO"|"STEP"|"PASS"|"FAIL"|"WARN"|"HEAL"|"API"|"DATA", msg, ts }
step
Step start/end
{ type: "step", status: "running"|"passed"|"failed", title, duration?, error?, ts }
heal
Selector healed
{ type: "heal", original, healed, strategy: "aria"|"testid"|"css"|"xpath", ts }
complete
Run finished
{ type: "complete", status: "passed"|"failed", passed, failed, duration, ts }
error
Crash
{ type: "error", msg, ts }
stopped
Manual stop
{ type: "stopped", ts }

Client → Server

ws.send(JSON.stringify({ type: "stop" }));  // abort active run

Full Handler Example

function handleEvent(event) {
  switch (event.type) {
    case 'log':
      appendToLogPanel(event.level, event.msg, event.ts); break;
    case 'step':
      updateStepStatus(event.title, event.status, event.duration);
      if (event.error) showStepError(event.title, event.error); break;
    case 'heal':
      showHealNotification(event.original, event.healed, event.strategy);
      incrementHealCounter(); break;
    case 'complete':
      showRunSummary(event.status, event.passed, event.failed, event.duration);
      ws.close(); break;
    case 'error':
      showError(event.msg); ws.close(); break;
  }
}

Quick Start

Get the full stack running in under 10 minutes

1 — Install server dependencies

cd playarc-server
npm install
npx playwright install    # downloads Chromium, Firefox, WebKit

2 — Start the server

node server.js

╔════════════════════════════════════════╗
║  PlayArc Server  →  localhost:4000     ║
║  REST   http://localhost:4000/api      ║
║  WS     ws://localhost:4000/ws         ║
╚════════════════════════════════════════╝

3 — Open the apps

Open ProjectManager.html to manage projects. Click 📁 My Projects to see the pre-seeded Pizza HQ demo project. Click ▶ Open on it to launch PlayWriteDemo.html with that project loaded.

Alternatively, open PlayWriteDemo.html directly — it will auto-load whatever project was last set active in the Project Manager.

⚠️
CORS note: The server allows all origins (*) for development. Lock this down before exposing to a network.

4 — Import a Katalon project

In Project Manager, go to 📥 Import Katalon. Drop your Katalon project ZIP file onto the upload zone. The engine will analyse all .rs, .groovy, .glbl, .csv and Libs/ files automatically, show a preview with heatscores per object, then create a fully-populated PlayArc project on confirmation.

5 — Run a test suite

# In the UI: open PlayArcDemo → Runner tab → ▶ Run
# Or directly via curl:
curl -X POST http://localhost:4000/api/suites/1/run \
     -H "Content-Type: application/json" \
     -d '{"browser":"chromium","headless":true}'

# Returns:
{ "runId": "run_a3f2b8c1", "status": "queued", "suite": "Login Validation" }

6 — Environment variables

PORT=4000               # server port (default 4000)
DB_PATH=./playarc.db    # SQLite file location
TESTS_DIR=./generated-tests   # where .spec.ts files are written
RESULTS_DIR=./test-results    # Playwright output
SHOTS_DIR=./screenshots       # screenshot storage
